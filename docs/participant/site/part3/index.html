    Part 3: 動作確認とレビュー

    所要時間: 10分

    このパートでは、Part 2で追加した機能が正しく動作するか確認し、IBM Bobのコードレビュー機能を使います。

    🎯 このパートのゴール

    • 追加した機能をテストする
    • IBM Bobのコードレビュー機能を使う
    • コードの品質を確認する

    ステップ1: 追加した機能をテストする（5分）

    Part 2で追加した3つの機能をテストします。

    テスト1: 商品画像の表示

    目的

    検索結果に商品画像が表示されることを確認します。

    手順

    1. Swagger UIを開く 

      http://localhost:8000/docs

    2. /searchエンドポイントを開く

    3. 緑色の「POST」ボタンの横の「/search」をクリック

    4. 「Try it out」をクリック

    5. 以下を入力 

      {
  "query": "赤いスニーカー"
}

    6. 「Execute」をクリック

    7. 結果を確認 

      {
  "results": [
    {
      "product_name": "赤いランニングシューズ",
      "image_url": "https://example.com/images/red-shoes.jpg",
      "similarity_score": 0.92,
      "price": 8900
    }
  ]
}

    確認ポイント

    • ✅ image_urlフィールドが存在する
    • ✅ URLが正しい形式（https://で始まる）
    • ✅ すべての検索結果に画像URLが含まれる

    テスト2: 価格フィルター

    目的

    価格帯で絞り込みができることを確認します。

    手順

    1. Swagger UIで/searchを開く

    2. 「Try it out」をクリック

    3. 以下を入力 

      {
  "query": "スニーカー",
  "filters": {
    "price_range": {
      "min": 5000,
      "max": 10000
    }
  }
}

    4. 「Execute」をクリック

    5. 結果を確認

    6. すべての商品の価格が5000〜10000円の範囲内

    確認ポイント

    • ✅ 指定した価格範囲の商品のみ表示される
    • ✅ 範囲外の商品は表示されない
    • ✅ セマンティック検索も機能している

    色々な価格帯を試す

    安い商品: 

    {
  "query": "ノートパソコン",
  "filters": {
    "price_range": {
      "min": 0,
      "max": 50000
    }
  }
}

    高級商品: 

    {
  "query": "カメラ",
  "filters": {
    "price_range": {
      "min": 100000,
      "max": 300000
    }
  }
}

    テスト3: レコメンド理由の表示

    目的

    なぜその商品がレコメンドされたのか、理由が表示されることを確認します。

    手順

    1. Swagger UIで/searchを開く

    2. 「Try it out」をクリック

    3. 以下を入力 

      {
  "query": "初心者向けのカメラ"
}

    4. 「Execute」をクリック

    5. 結果を確認 

      {
  "results": [
    {
      "product_name": "エントリーモデル デジタルカメラ",
      "similarity_score": 0.88,
      "reason": "検索クエリ「初心者向けのカメラ」と以下の点で類似しています：",
      "similar_features": [
        "対象: 初心者・入門者向け",
        "カテゴリ: デジタルカメラ",
        "特徴: 簡単操作、オートモード充実"
      ]
    }
  ]
}

    確認ポイント

    • ✅ reasonフィールドが存在する
    • ✅ similar_featuresに類似点がリスト表示される
    • ✅ 理由が分かりやすい日本語で書かれている

    色々な検索で理由を確認

    ビジネス向け: 

    {
  "query": "ビジネス向けのノートパソコン"
}

    高性能: 

    {
  "query": "高性能なゲーミングPC"
}

    ✅ テスト完了チェック

    • 商品画像が表示される
    • 価格フィルターが動作する
    • レコメンド理由が表示される
    • すべての機能が正しく動作する

    ステップ2: IBM Bobのコードレビュー（5分）

    IBM Bobのコードレビュー機能を使って、コードの品質を確認します。

    2-1. コードレビューとは？

    コードレビュー = コードの品質をチェックすること

    確認すること: - コードが読みやすいか - セキュリティの問題はないか - パフォーマンスの問題はないか - ベストプラクティスに従っているか

    2-2. /reviewコマンドを実行

    ステップ1: レビューしたいファイルを開く

    1. IBM Bob IDEの左側のファイルツリーを確認

    2. app.pyをクリック

    3. ファイルが開きます

    4. ファイルの内容が表示される

    5. Part 2で変更したコードが含まれています

    ステップ2: /reviewコマンドを実行

    1. IBM Bobのチャット入力欄をクリック

    2. 以下を入力してEnter 

      /review

    3. IBM Bobがコードを分析

    4. 10〜30秒程度かかります

    ステップ3: レビュー結果を確認

    IBM Bobが以下のような結果を表示します：

    ✅ コードレビュー完了

【良い点】
✓ API設計がRESTfulになっている
✓ エラーハンドリングが適切
✓ コメントが充実している
✓ 型ヒントが使用されている

【改善提案】
⚠️ パフォーマンス
  - ベクトル検索結果のキャッシュを検討
  - 大量リクエストへの対策が必要

⚠️ セキュリティ
  - APIキーを環境変数化（既に実装済み）
  - 入力値のバリデーション強化を推奨

⚠️ 保守性
  - ログ出力の追加を推奨
  - テストケースの追加を検討

    2-3. レビュー結果の読み方

    良い点（✓）

    IBM Bobが評価したコードの良い点です。

    例: - ✓ RESTful API設計 → 業界標準に準拠 - ✓ 型ヒントの使用 → コードが読みやすい - ✓ エラーハンドリング → 堅牢な実装

    改善提案（⚠️）

    IBM Bobが提案する改善点です。

    カテゴリ: - パフォーマンス: 処理速度の最適化 - セキュリティ: セキュリティリスクの軽減 - 保守性: コードの読みやすさ、メンテナンスのしやすさ

    2-4. 改善提案への対応（オプション）

    時間があれば、改善提案に対応してみましょう。

    例: ログ出力の追加

    IBM Bobに以下のように依頼します：

    ログ出力を追加してください。
検索クエリと結果件数をログに記録してください。

    IBM Bobが自動的にコードを修正してくれます。

    • コードレビューの重要性を理解した

    🆘 困ったときは

    よくある質問

    Q1: テストが失敗する

    対処法: 1. アプリケーションが起動しているか確認 2. 正しいURLにアクセスしているか確認 3. 入力したJSONの構文が正しいか確認 4. 講師に相談

    Q2: /reviewコマンドが動作しない

    対処法: 1. ファイルが開かれているか確認 2. Codeモードになっているか確認 3. IBM Bobを再起動 4. 講師に相談

    Q3: レビュー結果が理解できない

    対処法: 1. IBM Bobに「この改善提案を詳しく説明してください」と質問 2. 講師に説明してもらう 3. 重要な部分だけ理解する
