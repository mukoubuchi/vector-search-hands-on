    3️⃣ Part 3: 動作確認とレビュー

    所要時間: 15 分

    このパートでは、Part 2 で実装した機能の動作確認と、IBM Bob のコードレビュー機能を使った品質向上を行います。

    🎯 このパートで学ぶこと

    • Swagger UI を使った API テスト方法
    • セマンティック検索の動作確認
    • フィルター機能の検証
    • IBM Bob の /review コマンドによるコードレビュー

    1. アプリケーションの動作確認（10 分）

    📝 事前確認

    動作確認を始める前に、以下を確認してください：

    1. アプリケーションが起動していること

    2. ターミナルで python app.py が実行中であること

    3. エラーメッセージが表示されていないこと

    4. Swagger UI にアクセスできること

    5. ブラウザで http://localhost:8000/docs を開く

    6. API ドキュメントが表示されること

    🔍 Swagger UI の使い方

    Swagger UI は、API を視覚的にテストできるツールです。以下の手順で操作します：

    基本操作

    1. エンドポイントを選択

    2. /search エンドポイントをクリック

    3. エンドポイントの詳細が展開される

    4. 「Try it out」ボタンをクリック

    5. 入力フィールドが編集可能になる

    6. パラメータを入力

    7. query フィールドに検索クエリを入力

    8. 必要に応じて filters を設定

    9. 「Execute」ボタンをクリック

    10. API が実行される

    11. レスポンスが下部に表示される

    12. 結果を確認

    13. Response body セクションで JSON レスポンスを確認

    14. Response headers でヘッダー情報を確認

    🧪 テストケース 1: 基本的な商品検索

    目的

    Part 2 で実装した「商品画像と類似度スコアの表示」機能を確認します。

    手順

    1. Swagger UI で /search エンドポイントを開く

    2. 「Try it out」をクリック

    3. 以下のパラメータを入力

    {
  "query": "赤いスニーカー"
}
    1. 「Execute」をクリック

    確認ポイント

    レスポンスの JSON を確認し、以下が含まれていることを確認します：

    {
  "results": [
    {
      "product_name": "商品名",
      "image_url": "https://...",           // ✅ 画像 URL が表示される
      "similarity_score": 0.92,             // ✅ 類似度スコアが表示される
      "price": 8900,
      "category": "スニーカー"
    }
  ]
}

    期待される結果:

    • ✅ image_url フィールドに画像 URL が含まれている
    • ✅ similarity_score が 0.0〜1.0 の範囲で表示される
    • ✅ スコアが高い順（類似度が高い順）に並んでいる

    トラブルシューティング

    画像が表示されない場合:

    • image_url フィールドが存在するか確認
    • URL が正しい形式か確認（https:// で始まる）
    • サンプルデータに画像 URL が含まれているか確認

    類似度スコアが表示されない場合:

    • similarity_score フィールドが存在するか確認
    • Part 2 の実装が正しく反映されているか確認
    • アプリケーションを再起動してみる

    🧪 テストケース 2: フィルター機能の確認

    目的

    Part 2 で実装した「価格帯とカテゴリのフィルター」機能を確認します。

    手順 1: 価格帯フィルター

    1. Swagger UI で /search エンドポイントを開く

    2. 「Try it out」をクリック

    3. 以下のパラメータを入力

    {
  "query": "スニーカー",
  "filters": {
    "price_range": {
      "min": 5000,
      "max": 10000
    }
  }
}
    1. 「Execute」をクリック

    確認ポイント（価格帯フィルター）

    レスポンスを確認し、以下を検証します：

    {
  "results": [
    {
      "product_name": "商品 A",
      "price": 7500,    // ✅ 5000〜10000 の範囲内
      "category": "スニーカー"
    },
    {
      "product_name": "商品 B",
      "price": 8900,    // ✅ 5000〜10000 の範囲内
      "category": "スニーカー"
    }
  ]
}

    期待される結果:

    • ✅ すべての商品の価格が 5000〜10000 円の範囲内
    • ✅ 範囲外の商品は表示されない
    • ✅ セマンティック検索も機能している（「スニーカー」に関連する商品）

    手順 2: カテゴリフィルター

    1. 以下のパラメータを入力
    {
  "query": "赤い",
  "filters": {
    "category": "スニーカー"
  }
}
    1. 「Execute」をクリック

    確認ポイント（カテゴリフィルター）

    {
  "results": [
    {
      "product_name": "赤いランニングシューズ",
      "category": "スニーカー",    // ✅ カテゴリが一致
      "color": "赤"                // ✅ セマンティック検索も機能
    }
  ]
}

    期待される結果:

    • ✅ すべての商品のカテゴリが「スニーカー」
    • ✅ 「赤い」という検索クエリも考慮されている
    • ✅ 他のカテゴリの商品は表示されない

    手順 3: 複合フィルター

    1. 価格帯とカテゴリを同時に指定
    {
  "query": "ビジネス向け",
  "filters": {
    "category": "ノートパソコン",
    "price_range": {
      "min": 100000,
      "max": 200000
    }
  }
}
    1. 「Execute」をクリック

    確認ポイント（複合フィルター）

    期待される結果:

    • ✅ カテゴリが「ノートパソコン」
    • ✅ 価格が 100000〜200000 円の範囲内
    • ✅ 「ビジネス向け」という意味も考慮されている

    トラブルシューティング

    フィルターが機能しない場合:

    • JSON の構文が正しいか確認（カンマ、括弧の位置）
    • フィルター条件がサンプルデータに存在するか確認
    • Part 2 の実装が正しく反映されているか確認

    結果が 0 件の場合:

    • フィルター条件が厳しすぎる可能性
    • サンプルデータに該当する商品があるか確認
    • 価格帯やカテゴリ名を調整してみる

    🧪 テストケース 3: レコメンデーション理由の確認

    目的

    Part 2 で実装した「レコメンデーション理由の表示」機能を確認します。

    手順

    1. Swagger UI で /search エンドポイントを開く

    2. 以下のパラメータを入力

    {
  "query": "初心者向けのカメラ"
}
    1. 「Execute」をクリック

    確認ポイント

    レスポンスに以下の情報が含まれていることを確認します：

    {
  "results": [
    {
      "product_name": "エントリーモデル デジタルカメラ",
      "similarity_score": 0.88,
      "reason": "検索クエリ「初心者向けのカメラ」と以下の点で類似しています：",
      "similar_features": [
        "対象: 初心者・入門者向け",
        "カテゴリ: デジタルカメラ",
        "特徴: 簡単操作、オートモード充実"
      ]
    }
  ]
}

    期待される結果:

    • ✅ reason フィールドに説明文が表示される
    • ✅ similar_features に具体的な類似点がリスト表示される
    • ✅ ユーザーが納得できる説明になっている

    セマンティック検索の確認

    以下のような「意味」を理解した検索ができているか確認します：

    検索クエリ 期待される理解
    「初心者向け」 → 「入門」「簡単」「ビギナー」
    「ビジネス向け」 → 「仕事」「オフィス」「プロフェッショナル」
    「高性能」 → 「ハイスペック」「高速」「パワフル」

    トラブルシューティング

    理由が表示されない場合:

    • reason フィールドが実装されているか確認
    • Part 2 の実装を見直す
    • IBM Bob に「理由が表示されない」と相談

    理由が不自然な場合:

    • 埋め込みモデルの精度の問題の可能性
    • サンプルデータの説明文を確認
    • より具体的な検索クエリを試す

    📊 動作確認のまとめ

    すべてのテストケースを実行したら、以下のチェックリストで確認します：

    • 基本検索: 商品画像と類似度スコアが表示される
    • 価格帯フィルター: 指定した価格範囲の商品のみ表示される
    • カテゴリフィルター: 指定したカテゴリの商品のみ表示される
    • 複合フィルター: 複数の条件を組み合わせて検索できる
    • レコメンデーション理由: 類似点が具体的に表示される
    • セマンティック検索: 意味を理解した検索ができる

    2. IBM Bob によるコードレビュー（5 分）

    🎯 コードレビューの目的

    IBM Bob の /review コマンドを使用して、以下を確認します：

    • コードの品質（可読性、保守性）
    • パフォーマンスの問題
    • セキュリティリスク
    • ベストプラクティスへの準拠

    📝 /review コマンドの実行手順

    ステップ 1: IBM Bob を開く

    1. IBM Bob IDE のチャット画面を開く

    2. 画面下部のチャット入力欄をクリック

    3. 現在のモードを確認

    4. 右下の「Mode」セレクターを確認

    5. 「Code」モードになっていることを確認

    ステップ 2: レビュー対象ファイルを開く

    1. レビューしたいファイルを開く

    2. 例: app.py、search.py など

    3. Part 2 で編集したファイルを開く

    4. ファイルが IBM Bob に認識されていることを確認

    5. ファイルタブが表示されている

    6. コードが表示されている

    ステップ 3: /review コマンドを実行

    1. チャット入力欄に以下を入力
    /review

    1. Enter キーを押す

    2. IBM Bob がコードを分析するまで待つ

    3. 通常 10〜30 秒程度

    4. 「Analyzing code...」のようなメッセージが表示される

    📋 レビュー結果の確認

    IBM Bob は以下のような形式でレビュー結果を表示します：

    レビュー結果の例

    ✅ コードレビュー完了

【良い点】
✓ API エンドポイントが RESTful な設計になっている
✓ エラーハンドリングが適切に実装されている
✓ 型ヒント（Type Hints）が使用されている
✓ ドキュメント文字列（Docstrings）が充実している

【改善提案】
⚠️ パフォーマンス
  - ベクトル検索結果のキャッシュ機構の追加を検討
  - 大量のリクエストに対する負荷対策が必要

⚠️ セキュリティ
  - API キーの環境変数化を推奨
  - 入力値のバリデーション強化

⚠️ 保守性
  - ログ出力の強化（デバッグ情報の追加）
  - テストケースの追加を推奨

    🔍 レビュー結果の読み方

    1. 良い点（✓）

    IBM Bob が評価したコードの良い点です。以下のような観点で評価されます：

    • 設計: API 設計、アーキテクチャ
    • 実装: コードの書き方、パターンの使用
    • ドキュメント: コメント、説明の充実度

    例:

    • ✓ RESTful な API 設計 → 業界標準に準拠している
    • ✓ 型ヒントの使用 → コードの可読性が高い
    • ✓ エラーハンドリング → 堅牢な実装

    2. 改善提案（⚠️）

    IBM Bob が提案する改善点です。以下のカテゴリに分類されます：

    パフォーマンス

    意味: 処理速度やリソース使用量の最適化

    例:

    • キャッシュ機構の追加
    • データベースクエリの最適化
    • 不要な処理の削減

    対応方法:

    # 改善前
results = search_vector_db(query)  # 毎回検索

# 改善後（キャッシュ使用）
if query in cache:
    results = cache[query]
else:
    results = search_vector_db(query)
    cache[query] = results
    セキュリティ

    意味: セキュリティリスクの軽減

    例:

    • API キーのハードコード回避
    • 入力値のバリデーション

    3. よくある質問とトラブルシューティング

    🤔 よくある質問（FAQ）

    Q1: Swagger UI にアクセスできない

    症状: http://localhost:8000/docs にアクセスしても表示されない

    原因と対処法:

    1. アプリケーションが起動していない
    # ターミナルで確認
python app.py

    • 「Running on http://localhost:8000」と表示されるか確認

    • ポートが使用中

    # 別のポート（例: 8001）で起動
python app.py --port 8001

    • http://localhost:8001/docs でアクセス

    • ファイアウォールの問題

    • セキュリティソフトがポート 8000 をブロックしている可能性

    • 一時的に無効化するか、ポートを許可

    Q2: 検索結果が 0 件になる

    症状: どんなクエリでも結果が返ってこない

    原因と対処法:

    1. サンプルデータが投入されていない
    # データ投入スクリプトを実行
python scripts/load_sample_data.py

    1. Milvus への接続エラー

    2. ターミナルのエラーメッセージを確認

    3. TechZone 環境の接続情報を再確認

    4. 埋め込みモデルの問題

    5. watsonx.ai の API キーが正しいか確認

    6. API の利用制限に達していないか確認

    Q3: 類似度スコアが表示されない

    症状: 検索結果は返るが、similarity_score が含まれていない

    原因と対処法:

    1. Part 2 の実装が反映されていない

    2. アプリケーションを再起動

    3. コードの変更が保存されているか確認

    4. レスポンスの構造を確認

    # app.py で以下を確認
return {
    "results": [
        {
            "similarity_score": score,  # この行があるか
            ...
        }
    ]
}

    Q4: フィルターが機能しない

    症状: フィルター条件を指定しても、すべての商品が表示される

    原因と対処法:

    1. JSON の構文エラー
    // ❌ 間違い（カンマが多い）
{
  "query": "スニーカー",
  "filters": {
    "price_range": {
      "min": 5000,
      "max": 10000,
    }
  }
}

// ✅ 正しい
{
  "query": "スニーカー",
  "filters": {
    "price_range": {
      "min": 5000,
      "max": 10000
    }
  }
}

    1. フィルター実装の確認

    2. Part 2 でフィルター機能を実装したか確認

    3. IBM Bob に「フィルターが機能しない」と相談

    Q5: /review コマンドが動作しない

    症状: /review を実行してもレビュー結果が表示されない

    原因と対処法:

    1. ファイルが開かれていない

    2. レビュー対象のファイルを IBM Bob で開く

    3. ファイルタブが表示されているか確認

    4. モードが間違っている

    5. 右下の「Mode」セレクターを確認

    6. 「Code」モードに切り替える

    7. IBM Bob の再起動

    8. IBM Bob IDE を再起動

    9. プロジェクトフォルダを再度開く

    🔧 トラブルシューティングの基本手順

    問題が発生した場合、以下の順序で確認します：

    ステップ 1: エラーメッセージを確認

    1. ターミナルのエラーメッセージを読む

    2. エラーの種類を特定

    3. エラーが発生した行番号を確認

    4. ブラウザの開発者ツールを確認

    5. F12 キーで開発者ツールを開く

    6. Console タブでエラーを確認

    ステップ 2: 基本的な確認

    • アプリケーションが起動しているか
    • 正しい URL にアクセスしているか
    • インターネット接続は正常か
    • TechZone 環境にアクセスできるか

    ステップ 3: IBM Bob に相談

    問題が解決しない場合、IBM Bob に相談します：

    「検索結果が 0 件になります。エラーメッセージは以下の通りです：
[エラーメッセージをコピー]」

    IBM Bob が原因を分析し、解決策を提案してくれます。
