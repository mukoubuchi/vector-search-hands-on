# Worklog 記載ルール

このドキュメントでは、`worklog.md`の記載方法について説明します。

## 基本ルール

### 1. 時系列の順序

**重要**: worklogは**先頭に行くほど日時が新しくなる**ように記載します。

- 新しいエントリは常にファイルの先頭に追加
- 古いエントリは下に残る
- 最新の作業内容が最初に目に入るようにする

### 2. エントリの構造

各エントリは以下の構造で記載します：

```markdown
## YYYY年MM月DD日 HH:MM JST - 作業タイトル

### 作業内容

作業の概要を簡潔に記載

### 実施した対応

1. **項目1**
   - 詳細な説明
   - 具体的な手順

2. **項目2**
   - 詳細な説明
   - コマンド例など

### 成果

- 達成したこと
- 作成/更新したファイル
- その他の成果物

**完了日時**: YYYY年MM月DD日 HH:MM JST
```

## 記載例

### 良い例

```markdown
## 2026年5月28日 17:28 JST - vector-search-builder.zipの更新

### 作業内容

Vector Search Builderモードの配布パッケージを最新の状態に更新しました。

### 実施した対応

1. **zipファイルの更新**
   - 既存のvector-search-builder.zipを最新のファイルで更新
   - 以下のファイルを含む:
     - `.bob/custom_modes.yaml`: カスタムモード定義
     - `.bob/rules-vector-search-builder/`: ルールファイル

### 成果

- 全9ファイルが最新の状態でzipに含まれる
- 配布準備完了

**完了日時**: 2026年5月28日 17:28 JST
```

## 注意事項

1. **日時の形式**: `YYYY年MM月DD日 HH:MM JST`
2. **タイムゾーン**: 常にJST（日本標準時）を使用
3. **見出しレベル**: 
   - エントリタイトル: `##`
   - セクション: `###`
   - サブセクション: `####`（必要に応じて）
4. **コードブロック**: コマンドやコード例は必ずコードブロックで囲む
5. **箇条書き**: `-`または`1.`を使用して整理

## 更新手順

新しいエントリを追加する際は：

1. ファイルの**先頭**に移動
2. 新しいエントリを追加
3. 既存のエントリはそのまま残す
4. コミット＆プッシュ

**重要**: worklog.mdを更新したら、**必ずその後にGitへpush**すること。これにより、作業履歴が常に最新の状態でリモートリポジトリに保存されます。

## 参考

実際のworklog.mdを参照して、記載スタイルを確認してください。

# Zipファイル作成ルール

## 除外すべきファイル

zipファイルを作成する際は、以下のファイルを**絶対に含めてはいけません**：

### 機密情報を含むファイル
- `.env` - 環境変数ファイル（APIキー、パスワード等を含む）
- `*.key` - 秘密鍵ファイル
- `*.pem` - 証明書ファイル
- `*.p12` - 証明書ファイル
- `credentials.json` - 認証情報ファイル
- `secrets.*` - シークレット情報ファイル

### システムファイル
- `__pycache__/` - Pythonキャッシュディレクトリ
- `*.pyc` - Pythonコンパイル済みファイル
- `.DS_Store` - macOSシステムファイル
- `Thumbs.db` - Windowsシステムファイル
- `node_modules/` - Node.jsパッケージディレクトリ
- `.venv/`, `venv/` - Python仮想環境ディレクトリ

## Zipファイル作成コマンド例

```bash
# vector-search-builder.zipの作成例
zip -r vector-search-builder.zip \
  .bob/custom_modes.yaml \
  .bob/rules-vector-search-builder/ \
  setup/participant/ \
  -x "*.env" \
  -x "*/__pycache__/*" \
  -x "*.pyc" \
  -x "*/.DS_Store" \
  -x "*/node_modules/*" \
  -x "*/.venv/*" \
  -x "*/venv/*"
```

## 重要な注意事項

1. **機密情報の漏洩防止**: `.env`ファイルには本番環境の認証情報が含まれる可能性があるため、絶対に配布してはいけません
2. **サンプルファイルの提供**: 代わりに`.env.example`を含めること
3. **作成後の確認**: zipファイル作成後は必ず`unzip -l`で内容を確認すること
4. **自動化**: 可能であれば、除外ルールを含むスクリプトを作成して自動化すること

## チェックリスト

zipファイルを作成・配布する前に以下を確認：

- [ ] `.env`ファイルが含まれていないか
- [ ] その他の機密情報ファイルが含まれていないか
- [ ] システムファイルが含まれていないか
- [ ] `.env.example`が含まれているか
- [ ] `unzip -l`で内容を確認したか
